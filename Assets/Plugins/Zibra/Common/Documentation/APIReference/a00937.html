Detailed Description

Main ZibraLiquid component.

Each ZibraLiquid component corresponds to one instance of simulation. Different instances of simulation can't interact with each other.

Some parameters can't be changed after simulation has started and we created GPU buffers. Normally, simulation starts in playmode in OnEnable and stops in OnDisable. To change those parameters in runtime you want to have this component disabled, and after setting them, enable this component.

Liquid may run in the edit mode, specifically when you use initial state baking. In that case, you can't modify some parameters in edit mode too.

OnEnable will allocate GPU buffers, which may cause stuttering. Consider enabling liquid on level load, but with simulation/render paused, to not pay the cost of liquid initialization during gameplay.

Disabling liquid will free GPU buffers. This means that liquid state will be lost.

Various parameters of the liquid are spread throught multiple components. This is done so you can use Unity's Preset system to only change part of parameters.

◆ InitializeSimulation()

void InitializeSimulation ( )
inline

Initializes liquid simulation resources.

This is automatically called in OnEnable() if not in edit mode. To run liquid simulation in edit mode, you need to call it manually.

On success, sets Initialize to true.

On fail, cleans up simulation resources and throws an Exception.

Initialization allocates GPU resources, so calling this at runtime may cause stutter. Prefer to initialize liquid on scene load.

Has no effect if liquid is already initialized.

◆ ReleaseSimulation()

void ReleaseSimulation ( )
inline

Releases liquid simulation resources.

This is automatically called in OnDisable(). When running liquid simulation in edit mode, you may want to call it manually.

Sets Initialize to false.

Releases GPU resources and so frees up VRAM.

Has no effect if liquid is not initialized.

◆ MaxNumParticles

Maximum number of particles simulation may have.

Directly corresponds to maximum volume of liquid simulation may have.

Has noticeable VRAM impact.

Having more active particles in the simulation has noticeable performance impact.

◆ EnableDownscale

bool EnableDownscale = false

If enabled, makes liquid render in lower resolution.

Enabling downscale can significantly improve performance on mobile, by having way less pixels calculate pixel shader for the liquid.

Has no effect in Unity Render mode.

◆ DownscaleFactor

float DownscaleFactor = 0.5f

Factor of resolution downscale.

Lower factor corresponds to better performance, but lower visual quality.

If you set this value too high, you may get lower performance compared to downscale disabled. This is due to fact that we need to do additional pass to upscale liquid, so when resolution downscale is too high, performance win from lower shading resolution can potentially be less than performance loss due to cost of doing upscale pass. That's why value of 1.0 is not allowed and you have to disable downscale for full resolution.

Has no effect in Unity Render mode or when EnableDownscale is disabled.

◆ CurrentRenderingMode

RenderingMode CurrentRenderingMode = RenderingMode.MeshRender

Rendering mode of the liquid.

You can choose between:

  • Mesh Render mode - mode in which we: Generate mesh from the liquid. Render it with DrawIndirect in Native Plugin (optionally) Do raymarching pass to calculate light bounching inside the liquid in Native plugin. Shading inside Unity with customizeable shader. (optionally) Upscale pass to allow shading in lower resolution.
  • Unity Render mode - mode in which we: Generate mesh from the liquid. Copy it to Unity's Mesh Renderer. And Unity takes care of rendering that mesh.

To use Unity Render mode you'll need your own shader for liquid to render with.

In Unity Render mode you won't have raymarching results, so visual quality will be lower.

Unity Render mode has slight performance penalty, as it currently can not draw variable number of indices.

In Unity Render, material parameters set in liquid object have no effect. Since liquid can not control arbitrary material that may be set to render the liquid.

Mesh Render mode doesn't support VR at the moment, so will have to switch to Unity Render mode in order for VR to work.

Changing this parameter when liquid is initialized has no effect. During initialization we allocate different set of resources based on render mode. So changing render mode requires re-initialization.

See Documentation for more details.

Vector3 ContainerSize = new Vector3(10, 10, 10)

Liquid container size.

Liquid container is always a axis aligned box, and this Vector3 is sides of the box.

This indirectly affects performance, since aspect ratio of this box affects totan number of grid nodes. See GridNodeCount.

Liquid can not leave this box. You can, however, move this box. If you do that, liquid will try to stay in place in world space, unless EnableContainerMovementFeedback is disabled.

◆ Affine

ComputeBuffer [] Affine
get

Buffers containing affine velocity matrices, velocities and foaming values for each particle.

There are exactly 2 buffers, and the active one is flipped each simulation step.

This is a float4 buffer. Each pack of 4 float4's corresponds to particle. Format:

  • Affine[particleID * 4 + 0].xyz - 1st row of affine velocity matrix
  • Affine[particleID * 4 + 0].w - 1st particle specific random number
  • Affine[particleID * 4 + 1].xyz - 2nd row of affine velocity matrix
  • Affine[particleID * 4 + 1].w - 2nd particle specific random number
  • Affine[particleID * 4 + 2].xyz - 3rd row of affine velocity matrix
  • Affine[particleID * 4 + 2].w - 3rd particle specific random number
  • Affine[particleID * 4 + 3].xyz - Velocity
  • Affine[particleID * 4 + 3].w - Foaming value
